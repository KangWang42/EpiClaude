# 报告类 Word 文档写作 skill

> **一句话定位**：把已经成立的事实（分析结果 / 流程 / 进展）写成读者一眼能用、看不出 AI 痕迹的专业中文报告 docx。
> 报告 ≠ 论文：论文为审稿人证明严谨，报告为读者快速决策与执行服务。

---

## 〇、铁律（每次生成都适用，违反 = 未完成）

1. **数据有源、零编造**：所有数字（样本量、估计值、CI、P、百分比、日期、金额）必须取自项目结果文件
   （`07_paper/0_result_summaries.md`、`03_tables/` 导出表或用户明确给的来源），逐字一致、不四舍五入到与源不符。
   源里没有的 → 标 `[待确认]`，不要瞎填。
2. **学术书面语**：研究者/执行者视角（"本报告/我们完成了 X"），不用口语、网络词、AI 助手腔（"我建议你…""让我们…"）。
   标题用名词短语、不用反问；英文缩写首次出现给全称。
3. **零 AI 痕迹与零 emoji**：不堆开场闭场套话、不用 emoji / emdash、不出现 AI 工作痕迹字样
   （AI辅助 / assistant / 待人工复核 等）。中文文风过 `humanizer-zh` 的反 AI 黑名单意识。
4. **结论先行**：报告开头先给最重要的结论 / 要点 / 行动项，再展开依据。读者读前 1/4 就应抓住核心。
5. **面向行动**：凡涉及"下一步该怎么做"，写成可执行步骤（谁、做什么、按什么标准），不停留在描述。
6. **疑点先问**：读者对象 / 报告目的 / 关键口径 / 结论方向不明 → 先问用户，不擅自定调（同 CLAUDE.md §5）。
7. **局部修改也走完整标准**：哪怕只改一段，动后按 §五自检清单复扫，确保与全文术语、口径、编号一致。

## 一、报告 ≠ 论文（写之前先认清差别）

| 维度 | 论文（academic-publishing） | 报告（本 skill） |
|---|---|---|
| 读者 | 审稿人、同行 | 决策者、客户、团队、执行人 |
| 目的 | 证明结论可信、可发表 | 让读者快速理解并据以行动 |
| 结构 | IMRaD 固定、参考文献规范 | 灵活，按"读者需要先知道什么"组织 |
| 开头 | 背景铺垫 → 逐步推进 | 执行摘要 / 结论先行 |
| 语气 | 客观克制、回避主观 | 客观但更直接，可给明确建议 |
| 篇幅取向 | 完整论证 | 抓重点、去冗余、可要点化 |
| 引用 | 严格文献规范 | 注明来源即可，不强求文献格式 |

报告仍要：数据有源、书面语、零 AI 痕迹、零编造——这几条与论文一致，不放松。

## 二、报告结构（按报告类型取用，不必全有）

**通用骨架**（按需裁剪、重排）：

1. **标题页 / 抬头**：报告名（名词短语）、副标题、日期、版本 / 编制方（可选）。
2. **执行摘要 / 核心要点**（多数报告必备）：3–6 条，给最重要的结论、数字、行动项。读者只读这一段也不致误判。
3. **背景 / 目的**：为什么有这份报告、要解决什么问题、范围边界。
4. **方法 / 过程**（分析或操作类报告）：做了什么、依据什么标准。简明，不展开论文级细节。
5. **结果 / 发现**：核心内容。按主题分节，每节"先结论后依据"，配表 / 图 / 要点。
6. **建议 / 行动项**（决策类报告）：可执行步骤，必要时给"谁、何时、按什么标准"。
7. **附录**：明细表、原始数据指引、术语表、参考资料。

**按类型选骨架**：
- 分析 / 结果报告：执行摘要 → 背景目的 → 方法 → 发现（分主题）→ 结论建议 → 附录。
- 进展 / 工作报告：核心要点 → 已完成 → 进行中 → 待办与风险 → 下一步。
- 操作指南 / 手册（如填写指南、SOP）：用途与适用对象 → 核心概念 → 分步操作 → 结果解读 → 注意事项 / 常见问题。
- 说明 / 备忘：结论先行 → 关键事实 → 影响 → 需对方决策的点。

## 三、写作风格

- **结论先行**：每个章节、每个段落都先给判断 / 结论，再给支撑。避免"铺垫半页才到重点"。
- **一节一主题**：标题用名词短语点明该节结论（"低体重提示更差预后"优于"关于低体重的分析"）。
- **能要点化就要点化**：并列事实、步骤、清单用项目符号或编号；连续论证才用段落。
- **可操作**：动词开头写行动项（"核对…""填写…""提交…"），给明确判定标准而非含糊形容。
- **量化优先**：能给数字就给数字并注来源；避免"较多 / 明显 / 大幅"等无依据修饰。
- **读者语言**：少用内部变量名 / 脚本名 / 调参过程；必要的专业术语首次出现给一句解释。
- **专业中性、不哄读者**：标题与标签用中性名词，不用照顾式 / 哄人式措辞。禁用"（先读这一段）""一句话任务""本指南教你…""手把手""必须先弄清""三个你要知道的…"等预设读者水平、口语化的说法——直接写"核心要点""概念""范围""任务"即可。把读者当专业同行。

## 四、docx 排版规范（程序生成，统一用 python-docx）

**字体（中英分设，必须设 eastAsia）**：正文中文宋体 / 英文 Times New Roman 10.5pt；标题黑体加粗；
一级标题 14–17pt、二级 13pt、三级 11–12pt。统一一种中文字体，不混用。

```python
from docx.oxml.ns import qn
from docx.shared import Pt, RGBColor
def setfont(run, cn="宋体", en="Times New Roman", size=10.5, bold=False, color=None):
    run.font.size = Pt(size); run.font.bold = bold; run.font.name = en
    run._element.rPr.rFonts.set(qn("w:eastAsia"), cn)   # 关键：中文 eastAsia 字体
    if color: run.font.color.rgb = RGBColor(*color)
```

**版式**：
- 标题居中或左对齐统一；正文 1.15–1.3 倍行距；段后 3–6pt，不靠空行撑版面。
- 表格用三线表（顶线 / 表头下线 / 底线，无竖线、无内部横线）；表头加粗，内容左对齐 / 数字右对齐；
  **所有单元格内容垂直居中**（`cell.vertical_alignment = WD_CELL_VERTICAL_ALIGNMENT.CENTER`）。
- 短的"标签：内容"型要点（如核心要点）用加粗标签段落，**不用项目符号点**；只有并列同类项才用项目符号。
- 表 / 图有题注（表在上、图在下），编号连续；图按 `publication-figures` 规范另出，docx 内嵌 PNG。
- 要点用项目符号；多层级用编号，层级 ≤ 3。
- **全文字体一律黑色**（标题、正文、表头、副标题、注释一律纯黑 #000000）；**不用彩色字、不用灰字**；层级靠字号与加粗区分，不靠颜色。**禁止 emoji、禁止彩虹色、禁止默认灰底**。

**硬禁止**：emoji；emdash 当连接号；口语 / 网络词；AI 痕迹字样；无来源的"最佳 / 显著 / 证明"。

## 五、生成流程与自检清单

**流程**：定读者与目的 → 选骨架（§二）→ 取数（铁律 1）→ 逐节"结论先行"写 → 套排版（§四）→ 生成 docx → 自检 → 交付时先报告"已自检项"。

**完成前自检（不等用户挑错）**：
- [ ] 执行摘要 / 核心要点已在开头给出，读者读前段即抓住核心
- [ ] 所有数字有源且与来源逐字一致；无源处标 `[待确认]`
- [ ] 每节先结论后依据；标题为名词短语
- [ ] 行动项可执行（动词开头 + 判定标准）
- [ ] 中文书面语、零口语 / 零 AI 痕迹 / 零 emoji；术语首现有解释
- [ ] 中英字体分设且设了 eastAsia；标题 / 正文字号层级一致
- [ ] 表为三线表、有题注、编号连续；图按规范内嵌
- [ ] 已实际生成 docx 并 Read 核验（无乱码、无空表、无错位）
- [ ] 术语 / 口径 / 编号全文一致（局部修改也复扫）
